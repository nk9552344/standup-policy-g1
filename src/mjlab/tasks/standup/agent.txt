1. TASK DEFINITION
================================================================================

ORIGINAL TASK: The `standup` folder contained a RECOVERY policy:
  - Robot spawns in a random FALLEN state (face-down, face-up, on side)
  - Learns to get up from the ground
  - Then holds still once standing

CONVERSION GOAL: Change it to a STAY-STAND (balance) policy:
  - Robot ALWAYS spawns STANDING (upright at HOME_KEYFRAME)
  - Learns to BALANCE and not fall
  - Learns to RECOVER from external pushes while staying upright
  - A fall TERMINATES the episode immediately (no "get up again" phase)

All work lives under:
  vsls:/src/mjlab/tasks/standup/

2. ROBOT: UNITREE G1
================================================================================

  - 29 DOF total, 23 real actuated DOFs
  - Pelvis standing height: 0.793 m (from g1.xml: <body name="pelvis" pos="0 0 0.793">)
    This is the height when ALL JOINTS ARE AT ZERO (T-pose / fully extended)
  - HOME_KEYFRAME (the actual standing pose used in training):
      hip_pitch = -0.1 rad
      knee      =  0.3 rad (bent)
      ankle     = -0.2 rad
    With these bent knees, actual pelvis height is ~0.72-0.75 m at rest.
  - PD gains (from G1 constants / HoST reference):
      Hip:   kp=150, kd=4
      Knee:  kp=200, kd=6
      Ankle: kp=40,  kd=2
  - Why HOME_KEYFRAME instead of KNEES_BENT_KEYFRAME:
    KNEES_BENT (hip=-0.312, knee=0.669, ankle=-0.363) is a forward-leaning
    crouch. Gravitational ankle torque ≈ 20 N·m; ankle PD stiffness only
    ≈ 10.5 N·m/rad. Robot falls in the first few sim steps before any policy
    gradient exists. HOME_KEYFRAME has ankle torque ≈ 2 N·m — well within
    the stiffness range.

Action scale override in env_cfgs.py:
  joint_pos_action.scale = {k: v * 0.25 for k, v in G1_ACTION_SCALE.items()}
  Reason: G1_ACTION_SCALE is tuned for locomotion (knee ~0.35 rad per unit
  raw action). For stay-stand, at iter 0 the MLP outputs ~1.0 per dim, which
  means ~0.35 rad off default per joint before any learning. Multiplying by
  0.25 limits worst-case random action to ~0.09 rad/joint, so pose/upright
  rewards fire from iter 0.

3. REFERENCE: HoST (InternRobotics)
================================================================================

  URL: https://github.com/InternRobotics/HoST
  This is a humanoid standing policy for G1 (and others). Key design decisions
  we studied to diagnose our training failures:

  A. REWARD STRUCTURE — Multiplicative, not additive
     HoST multiplies all task rewards together instead of summing them:
       self.rew_buf[:, task_group_index] *= rew
     This means ALL task terms must be non-zero for any reward to flow.
     We use additive rewards (standard MjLab approach).

  B. UPWARD PULL FORCE BOOTSTRAP
     HoST applies an upward force to the robot's base during early training:
       force_tensor[:, base_indices, 2] = self.force  (e.g. 100 N)
     The curriculum reduces this force as the robot improves.
     This is a critical bootstrapping mechanism — our policy has NO equivalent.
     We do not apply any upward assist. This is the biggest gap vs HoST.

  C. NO ORIENTATION-BASED TERMINATION
     HoST does NOT terminate on height or uprightness for the ground
     standing policy. They terminate only on:
       - Contact forces on specific body parts (elbow, shoulder, waist, knee, hip)
       - Joint velocity overflow
       - Base velocity overflow
     Our `fell_over` term terminates on height + uprightness, which is
     correct in principle but requires careful threshold tuning.

  D. PUSH POLICY
     HoST: push_robots = False for the G1 ground standing policy.
     We: push_robot fires every 8-15s with ±0.3 m/s from iteration 0.
     During early training, this is fine IF episodes are long enough (>8s)
     for the push to fire. With the fell_over bug fixed and episodes now
     extending to full 20s, pushes will fire once per episode.

  E. OTHER HYPERPARAMETERS
     - init_noise_std = 0.8 (we use 0.3 after fix, was 0.1)
     - 6-step observation history (we use current step only)
     - Episode length: 10s (we use 20s)
     - action_scale = 1 with delta-position PD control (we use absolute)
     - entropy_coef = 0 (same as us)

5. TRAINING FAILURE DIAGNOSIS AND FIXES
================================================================================

After completing all code conversion, training was started and the policy
made NO progress. User confirmed all four failure symptoms:

  SYMPTOM 1: Episodes end almost instantly (< 1 second)
  SYMPTOM 2: fell_over fires on almost every episode
  SYMPTOM 3: Policy std growing (e.g. 0.1 → 0.3+) — learn_std was True
  SYMPTOM 4: All rewards near zero — policy is doing nothing

────────────────────────────────────────────────────────────────────────────────
ROOT CAUSE 1: fell_over thresholds too tight (MOST CRITICAL)
────────────────────────────────────────────────────────────────────────────────

BUG: env_cfgs.py had:
  cfg.terminations["fell_over"].params["min_height"] = _G1_MIN_STANDING_HEIGHT
  # _G1_MIN_STANDING_HEIGHT = 0.65

HOME_KEYFRAME actual pelvis height ≈ 0.72–0.75 m (with bent knees).
min_height = 0.65 is only 7–10 cm below standing height.

Result: random policy actions (even at std=0.1) perturb the joints enough
to drop the pelvis below 0.65 m within a handful of steps. Episodes terminate
in < 1 second. The policy NEVER receives any reward because every episode
ends before reaching any meaningful state.

Additionally: min_uprightness = 0.5 (inheriting from base config) means the
robot terminates at 60° of tilt. This is too aggressive for early training
where the robot hasn't learned any corrective response.

The base config (standup_env_cfg.py) has min_height = MIN_STANDING_HEIGHT = 0.5
which is better, but the G1 override raised it to 0.65 — exactly wrong.

FIX APPLIED:
  cfg.terminations["fell_over"].params["min_height"] = 0.45
  cfg.terminations["fell_over"].params["min_uprightness"] = 0.2

  0.45 m = 57% of standing height = clearly fallen, not just crouching
  min_uprightness = 0.2 allows ~80° of tilt before terminating

WHEN TO TIGHTEN AGAIN:
  Once the policy can reliably stand through full 20s episodes without
  terminating, tighten back toward:
    min_height → 0.55 (still gives recovery room)
    min_uprightness → 0.35 (still allows significant push recovery)

────────────────────────────────────────────────────────────────────────────────
ROOT CAUSE 2: learn_std=True recreates the exact failure loop we fixed before
────────────────────────────────────────────────────────────────────────────────

BUG: rl_cfg.py had learn_std=True (mistakenly set during Phase 3).

Mechanism:
  Near-zero rewards
  → noisy advantage estimates
  → PPO's surrogate gradient pushes log_std up (actions that happened to
    have slightly positive advantage increase both mean and std)
  → larger, more chaotic actions
  → robot falls more → shorter episodes → even less reward
  → repeat (catastrophic feedback loop)

The long comment in rl_cfg.py (about "std growing 0.1 → 0.18 → 0.28...")
was written specifically about this exact failure mode with the RECOVERY policy.
The stay-stand policy has the same structure (bounded exp rewards, sparse signal
in early training) and therefore has the exact same failure mode.

FIX APPLIED:
  learn_std: False  (removes log_std from the optimizer entirely)
  init_std: 0.3     (raised from 0.1 to give wider exploration coverage
                     without relying on learned std growth)

REASONING:
  learn_std=False fixes std at init_std for the entire training run.
  The policy improves ONLY by shaping the MEAN action, which is exactly
  what we want for a "hold default pose while resisting pushes" task.
  entropy_coef=0.0 means there is no push to grow std from entropy loss.

────────────────────────────────────────────────────────────────────────────────
ROOT CAUSE 3: Shaping penalties active before robot can stand
────────────────────────────────────────────────────────────────────────────────

BUG: body_ang_vel.weight=-0.05, angular_momentum.weight=-0.01,
action_rate_l2.weight=-0.01 were all set non-zero in Phase 3.

These are stability-shaping signals designed for a policy that ALREADY stands
well and we want to make smoother. Activating them from iteration 0, when the
robot can't stand at all (due to Bug 1), adds pure noise to the value function
target. The value function tries to fit a sum of terms where 3 of them are
always near-zero (robot can't earn positive rewards) and 3 are near-zero
penalties (robot falls too fast to accumulate penalties). This corrupts
gradient estimates.

FIX APPLIED:
  body_ang_vel.weight = 0.0
  angular_momentum.weight = 0.0
  action_rate_l2.weight = 0.0

WHEN TO RE-ENABLE:
  Once the robot can stand reliably (episode rewards non-zero and rising,
  avg episode length > 10s), re-enable with small weights:
    body_ang_vel.weight = -0.05
    angular_momentum.weight = -0.01
    action_rate_l2.weight = -0.01

7. TRAINING TUNING ROADMAP
================================================================================

PHASE A (current — bootstrapping):
  - Goals: robot can stand for full 20s without fell_over firing
  - Indicators of success:
    * avg episode length > 15s
    * upright reward > 3.0 (out of max 5.0)
    * pose reward > 5.0 (out of max 10.0)
    * fell_over rate < 20%
  - Active rewards: hold_still, upright, pose, dof_pos_limits, self_collision
  - Disabled: body_ang_vel, angular_momentum, action_rate_l2
  - Termination: fell_over (0.45m / 0.2 uprightness — loose)

PHASE B (stability shaping):
  - Trigger: PHASE A success indicators met
  - Actions:
    1. Re-enable body_ang_vel.weight = -0.05
    2. Re-enable angular_momentum.weight = -0.01
    3. Re-enable action_rate_l2.weight = -0.01
    4. Tighten fell_over: min_height → 0.55, min_uprightness → 0.35
  - Goal: robot actively damps oscillations and doesn't flap arms

PHASE C (push recovery):
  - Trigger: PHASE B stable, robot not falling from pushes
  - Actions:
    1. Tighten fell_over further: min_height → 0.60, min_uprightness → 0.5
    2. Optionally increase push strength: ±0.5 m/s
    3. Consider terrain curriculum progression
  - Goal: robot recovers from hard pushes and returns to HOME_KEYFRAME

FUTURE: Bootstrap mechanism (HoST-style upward pull force)
  - HoST applies 100N upward on the base, reduces via curriculum
  - If PHASE A is still stuck after hundreds of iterations, this may be
    needed. Requires a new EventTerm that applies a body force in MuJoCo.
  - Check if mjlab.envs.mdp.events has apply_body_force or similar.
  - The force should be applied only to the base/pelvis body.
  - Curriculum: start at 100N, reduce by (success_rate * step) toward 0.

8. WHAT TO TRY IF TRAINING IS STILL NOT LEARNING
================================================================================

In priority order:

1. CHECK ACTUAL FELL_OVER THRESHOLD CALIBRATION
   Run a zero-policy test (output all zeros): does the robot stand for 20s?
   If not, the HOME_KEYFRAME starting position is still unstable for the
   G1 actuator parameters. Fix: reduce ankle stiffness demands or adjust
   HOME_KEYFRAME z-position slightly.

2. CHECK EPISODE LENGTH DISTRIBUTION
   After fixes, are episodes now > 5s on average? If they are still < 2s,
   the fell_over thresholds may still be too tight. Lower min_uprightness
   to 0.1 or raise min_height threshold even lower.

3. ADD EXPLORATION HELP
   init_std=0.3 gives ±0.075 rad per knee per step (0.3 * 0.25 * 1.0).
   If the policy is stuck at a local zero (output all zeros = default pose),
   temporarily raise init_std to 0.5 to force more exploration.

4. HoST BOOTSTRAP (LAST RESORT)
   If after 1000+ iterations the rewards are still near zero:
   Add an upward force event that applies 50-100N to the pelvis body.
   The robot will start "levitating" slightly, making it easy to earn
   upright/pose rewards, then reduce the force as rewards improve.

5. OBSERVATION HISTORY
   HoST uses 6-step history. Our policy sees only the current step.
   Adding history (concatenate last 3-6 observations) gives the policy
   velocity information even if the IMU is noisy.

12. TRAINING FAILURE DIAGNOSIS — SESSION 3 (2026-06-21)
================================================================================

Training showed: rewards declining after 2.5k steps, crash at 3.8k.
User observation: "robot takes small actions to stay standing but sometimes
falls faster than doing nothing."

CHARTS OBSERVED:
  - Train/mean_reward: 12→14 (step 0-2.5k), then declining, crash 11→5 at 3.8k
  - Episode_Reward/upright: 0.05-0.3, very low and declining
  - Episode_Reward/pose: 0.5-1.0, declining after step 3k
  - Episode_Reward/hold_still: 0.01-0.03 (robot moving fast)
  - Policy/mean_std = 0.3 flat (learn_std=False working correctly)
  - Loss/entropy ≈ 6.0 flat (matches 29-dim Gaussian at std=0.3 analytically)

ROOT CAUSE 1 (CRITICAL): upright std too tight — gradient vanishes at operating tilt
────────────────────────────────────────────────────────────────────────────────

Gradient of upright reward for policy = w * kernel / std²
                                      = w * exp(-xy²/std²) / std²

With old std=√0.05=0.224, at 20° tilt (xy²=sin²20°=0.117):
  kernel = exp(-0.117/0.05) = 0.096    ← near-zero
  gradient-factor = 5.0 * 0.096 / 0.05 = 9.6

With new std=√0.117=0.342, at 20° tilt:
  kernel = exp(-0.117/0.117) = exp(-1) = 0.368
  gradient-factor = 10.0 * 0.368 / 0.117 = 31.4  ← 3.3× improvement

The old reasoning ("tighter std gives clearer gradient") was WRONG.
The gradient magnitude = kernel / std². A near-zero kernel gives near-zero
gradient — the tightest std is the worst choice when the kernel collapses
to zero at the robot's operating range.

ROOT CAUSE 2 (CRITICAL): pose weight (10) > upright weight (5) — inverted
────────────────────────────────────────────────────────────────────────────────

For a balance task, upright MUST dominate. With old weights:
- When robot tilts and policy makes corrective ankle action:
  * The ankle moves away from HOME_KEYFRAME → pose reward DECREASES
  * The tilt starts correcting → upright reward INCREASES
  * With w_pose=10 >> w_upright=5, pose penalty > upright benefit
  * Policy gradient DISCOURAGES the corrective action
- Result: policy learned to return joints to HOME_KEYFRAME (strong pose
  gradient) even when this destabilizes balance. "Actions make it fall
  faster" is exactly this: pose-tracking actions actively fighting balance.

Reference: Isaac Lab G1 locomotion uses flat_orientation_l2.weight=-1.0
(L2 PENALTY) and joint_deviation.weight=-0.05 to -0.1 (tiny secondary).
Orientation is never the PRIMARY reward in any known G1 balance/locomotion
policy — pose is always secondary.
